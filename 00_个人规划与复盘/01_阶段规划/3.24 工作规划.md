# 背景
    在逐步深入业务的过程中，入职规划的目标和路径已经不足以指导现有行动。在想做的业务锁定在用 ai 模拟真人进行互动，岗位对应 AI 策略产品经理

    业务进展阶段：
    技术线：从初级的 prompt - > 搭建代码框架，实现咨询工作流，将技术应用到具体场景（进行中）-> 初步测试 - > 开发拓展 skill 使其具备长期记忆，数据分析能力、用户分析能力。 - > 尝试商业化
    业务线：idea - > 目前在做价值验证，现在是探索可能性 - > 有可能性之后怎么和市场（业务）结合-毛老师最重要的工作【咨询侠，时间因素影响优先级，助教志愿者，大学信息差（产品版），跟随 or 先做局部蛋糕】。

    现有资源：
    1. 毛老师 ppt 文稿
    2. 直播答疑课的问答
    3. 助教积累的助教问答
    4. 大学信息差问答库

    现有落地技术部分：
    
    v1框架：
    1. 参照 langgraph 架构，设定多 agent，用工作流的方式模拟咨询过程。
        1. 其中 gateway 和 browser 两个agent 讨论后决定问题分流.
        2. 答疑模式分 consult 和 coach， consult 模式专注于给省流结论，coach 则是在多次追问（可以给选项）后，给出一个具体回答。
        3. 输出前， Guardrail 进行一轮质检，并且将有价值的长期记忆写入 memory 模块。

    现有架构主要问题：
    1. 响应时间过长，现选择的大模型本身具备 thinking 和推理功能，目前设计的架构使得一次回答实际上是至少 4 倍的生成时间（每次要用 ai 的部分都调用一次）。
    2. 用户 id 和 memory 部分未开发。
    3. 业务匹配度低，是未充分理解需求，开发出的第一版。

    v2 具体读研模块：
    1. 当前不再沿用 v1 的多 agent 串行架构，而是围绕单一场景做收敛：聚焦“是否有必要读研”这一类高频咨询问题，把毛老师的理想问答拆成可执行的对话状态机。
    2. 技术实现上，采用单次调用的 unified prompt 架构，把用户分型、阶段判断、知识检索注入和回复生成合并到一次模型调用中，降低响应时延。
    3. 模型输出被约束为固定结构：先输出 thinking，再输出 reasoning JSON，最后输出给用户的话术。其中 reasoning JSON 负责承载 user type、当前阶段、下一阶段、关键信号提取等结构化结果，便于后处理和调试。目前仍然需要花两倍时间，但后续可以通过优化模型选择（选小模型）来实现。
    4. 在业务逻辑上，已把咨询流程拆为 S0-S5：先做动机探查，再根据 3 类用户进入不同分支，随后进行路径价值解释、行动建议和追问处理，使回复不再是泛化答疑，而是接近真人咨询的递进式引导。
    5. 在效果保障上，已补上测试集和评测链路：一套测试推理层是否分型、转阶段正确；一套测试端到端回复是否符合预期，用来支持每次改 prompt 或改代码后的回归验证。


# 复盘

    回顾目标：
        上个月作为入职第一个月，核心目的是快速适应业务需要（同时结合自己想做的方向），确保后续实习过程有效成长。

    结果：
        第一周，和毛老师对齐入职期望，讲清楚技术路线，idea 得到肯定，后续再探索 idea 的价值有多大。
        第二周，由于还在适应协作方式，同提示词的方式做出的 demo 比较匆忙，明确了需求是，用户一次性给一个更加稳定的信息，ai 给出一个 60 分的基于用户信息定制的指导，不是回答问题的逻辑，而是快速做情景指导的逻辑。点出了我在过程中，整体是研发视角。
        第三周，利用多 agent 架构，做了一个多维度的 demo，涵盖的是大学信息差场景。demo 不合格的原因是依然没切入业务，训练模式出错。
        第四周，和毛老师沟通后，拆解出毛老师给的案例回答，以此为逻辑做了读研场景的 demo。

        整体来看，人与 ai 对话的策略，我很感兴趣同时有一定业务价值和后续发展前景。明确了 ai 策略产品作为具体的校招岗位。明确了后续的开发需要，能稳定推进业务。
    
    超出预期的地方：
        自己做 demo 的效率比之前预想的要高，在实习期内估计所有的业务场景化都能做完，可以涉及 skill 领域，对技术的热情也比较高（烦的可能只是找bug，对技术原理还挺感兴趣）。

    不符合预期的地方：
        产品思维依旧不算优秀，业务上的问题基本上是毛老师发现的（比如第一次 demo 不够情景化，第二次 demo 依然没有明确理想回答是什么样子。）实际上思考业务的部分是整个推进里耗时最久的部分。
    
    再来一次会怎么做？
        依然在入职沟通之后，和毛老师聊技术路线。但第二周的沟通需要更加明确目标，当时定下来的目标不完全可执行，而是讲了先给一个 60 分回答的逻辑导致我误判到大学信息差那个场景，这样可以省下来一周的折腾。

    上级评价：实际表现和工作状态，工作产出，整体是超预期的，问问大象招实习生等价于大厂招 sp的水平，因为我之前没有在互联网大厂没有实习过，所以自己对实习的表现期待过高了。大厂产品经理的 sp 无非是核心部门做相对核心的业务，工作的头半年都是执行好、交给新人的任务能不能有效完成就行了（算快的），产品新人比较少能参与一些价值的探索和讨论中，大部分企业都不会对员工有这么高的期待。
    
    不达预期的问题，首先是小公司+毛老师有经验能做这种层面的沟通，我现在对自己的标准是一些资深产品经理 or 独立负责人的要求（事情本身是高门槛），这些事情难点在于，理解一个典型的互联网商业项目是怎么回事，这部分事情和新人产品经理的事情是无关的（基准的部分就是做一些迭代，学一些用研等部分让上级满意）。

    创业公司是缩短了，现在对自己的期待的预期是非常高的创业预期。除非是大厂路线 or 小公司遇到毛老师这样上级 概率极低。期待管理目前比较部分，不达预期的部分是需要学习的部分，1～3 年非常快，商业部分大公司是高经理甚至总监才有机会训练的，在洋葱学院做 vp 之后才有概念。

    特质和想要的东西目前是匹配的，所以才会分配到目前的这些事情，目前还是这样的想法，特质部分是很内核的东西。没有耐心的事情是，如果要走第一条路，大厂路径，做到我想要做的状态，大厂可能永远没有机会（总监以上都没什么机会，制约非常多），这条路得非常有耐心，升迁非常顺利五到七年都不一定能达到。
    第二条路，可能得至少大厂五年，然后跳出来。
    第三部分，在小公司做范式切换，7-10 年，快一点是 5-7 年，快依然有条件，在目前理论上最优的平台，积累也需要时间；不要半路有太强的执念，做自己想做的事情至少是 2～3 年，这个是一个评估的节点。愿景上有时候可能会过于急躁，会控制一些比较大的想法。

# 规划
    1. 业务线：在开始某个具体模块开发之前，先确定标准回答集合，设定考核的关键指标，做测试集，使得每次更改完代码之后做一次版本回归测试。
    2. 技术线：主要是快速对业务线的需求做验证，次要是对某个具体的模块设计进行探索，比如说做好评测，尝试更小更快响应的本地模型等。

    ai 策略产品职能参考：

    主要职责：
    1. 负责将业务目标和用户需求转化为 AI Agent 产品策略，明确核心场景、能力边界和风险边界。
    2. 负责 AI 模拟真人互动场景下的产品设计，定义对话流程、追问机制、问题分流、失败兜底及人工接管策略。
    3. 负责设计 Agent 工作流，推动知识库、案例库、记忆、工具调用等能力在业务场景中的整合与应用。
    4. 负责搭建评测体系，定义任务分层、成功标准、测试集、错误分类和评分规则，持续跟踪系统效果。
    5. 负责建立数据与实验闭环，围绕任务完成率、追问率、有效建议率等核心指标开展策略优化和版本对比。
    6. 负责分析失败样本，识别问题属于模型、Prompt、检索、工作流或记忆等环节，并据此制定优化方案。
    7. 负责推进多角色协作，与业务、研发、算法、运营团队共同推动策略落地、上线和迭代。
    8. 负责沉淀 AI 产品策略方法论，形成可复用的评测、优化和工作流设计规范。
