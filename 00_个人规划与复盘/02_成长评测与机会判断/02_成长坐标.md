# 成长坐标

## 衡量坐标

三套尺子各管一件事，互不重叠：

- **八维能力** — 量什么（能力的横轴）
- **S0-S5 成长阶段** — 每个维度有多成熟（能力的纵轴）
- **P-level 大厂对标** — 你做的事在行业里对应什么层级（外部参照系）

### 八维能力定义

先把"产品判断"和"研发控制"分开。AI 产品的研发能力，不该再被粗暴合并成"方案设计 + AI 策略"这一项。真正的差别在于：你是在控制模型怎么理解任务，还是在控制模型看到什么信息，还是在控制模型输出之后系统怎么验收和推进。

| #   | 维度                      | 看什么                           |
| --- | ----------------------- | ----------------------------- |
| 1   | 产品定义                    | 做不做、切哪条主线、边界到哪                |
| 2   | 用户理解                    | 用户到底卡在哪一层，真问题和伪问题怎么分          |
| 3   | Prompt / Instruction 设计 | 如何把策略翻译成模型可执行的任务、角色和输出契约      |
| 4   | Context 设计              | 如何组织知识、状态、画像、记忆，让模型在这一轮看到对的信息 |
| 5   | Harness 设计              | 如何用状态机、校验、路由、重试、工具边界把概率系统关进流程 |
| 6   | Eval 与诊断                | 如何定义"好"，如何建测试集、拆错误类型、做回归闭环    |
| 7   | 项目推进                    | 如何把判断推进成可交付物，跨人协作和节奏控制        |
| 8   | 商业理解                    | 为什么值得做、替代谁、在哪创造价值             |

其中 3-6 才是 AI 产品研发的核心。Prompt 控任务策略，Context 控证据供给，Harness 控流程稳定，Eval 控质量判断。四者不是一回事。

### 成长阶段 S0–S5

每个维度都可以用这把尺子量。S 阶段衡量的是**你在该维度上的独立程度和决策可靠度**——不是"做过什么"，而是"能稳定做到什么"。

| 阶段 | 一句话 |
|------|--------|
| S0 新手执行 | 靠他人分配，不能区分该不该做 |
| S1 独立执行 | 能完成子模块，理解开始成形 |
| S2 高潜 Owner 预备 | 参与价值验证和标准定义，对结果负责而不只是动作 |
| S3 场景 Owner | 独立负责场景闭环，模糊性下可靠决策 |
| S4 业务定义者 | 定义做什么不做什么，影响他人标准 |
| S5 平台负责人 | 跨场景复制方法论，塑造组织能力 |

> **八维 × S 阶段 = 你的能力矩阵。** 后面"当前位置"用的都是这套坐标。

> **注 2：传统 P-level 在 AI 产品 Phase 3+ 系统性低估几项能力**。AI 产品打破了"产品形态稳定、PM 不碰实现"的前提——产品行为是概率性的，PM 必须写 prompt、搭评测、读源码做架构决策。这个角色两年前不存在，大厂还没有成熟的职级体系去容纳它。具体而言：
> - **评测体系**（传统 P6-P7）：绝大多数 AI 团队没有评测。你的三层评测（reasoning 15c/49t + E2E 15c + LLM-as-Judge）方法论成熟度已超过大量已融资 AI 团队。当前最有外部议价力的单项。
> - **源码→架构决策**（传统 P6-P7+架构师）：在 AI 产品里"选什么框架"直接决定"产品能做什么"，能做这件事的 PM 极少——需要同时具备读代码的能力和做产品判断的框架。
> - **Prompt 架构**（传统 P5-P6）：本质上是 AI 产品的"后端架构"——传统产品 PM 不碰，AI 产品 PM 必须 own。
> - **诚实的限定**：以上分布是粗估；"稀缺" ≠ "高溢价"——大多数公司还没想清楚这个岗位值多少钱；你在 Phase 3 有独立交付物（v2 评测），在 Phase 4-5 目前只有规划文档。

**一句话**：你独立做的事集中在 **P6-P7**，正在向 **P7-P8** 延伸；你被暴露在 **P8-P9** 的决策场景里但还不能独立拍板。在同龄人还在 P4-P5 层执行的阶段，这个跨度非常显著——但"被暴露"≠"已具备"。

你的稀缺性不来自单项深度，来自整合宽度——大多数人要么技术强不懂产品，要么产品强不碰代码。

### 八维能力水位

**整体判断：S2→S3 过渡区**

你已经在做 S4 甚至更高层级的事，但"做过"和"稳定具备"之间有差距。

| # | 维度 | 当前水位 | S 阶段 | 证据 / 说明 |
|---|------|---------|--------|-------------|
| 1 | 产品定义 | 能论证方案，不能独立定方向 | S2 | 方向毛老师已确认，你做方案设计和论证 |
| 2 | 用户理解 | 依赖毛老师框架 | S1→S2 | 关键洞察仍高度依赖毛老师输入 |
| 3 | Prompt 设计 | ★★★ 强 | S2→S3 | thinking + JSON + 用户回复三层结构，能约束模型输出为系统可消费格式 |
| 4 | Context 设计 | ★★ 中偏弱（最大短板） | S1→S2 | 理解知识/状态/画像/历史要进系统，但长期记忆、画像治理、检索策略未被独立验证透 |
| 5 | Harness 设计 | ★★★ 强 | S2→S3 | 状态机 + validator + 路由 + 回退 + 输出契约。非传统 PM 会碰的东西，当前 AI 产品里最被低估的研发能力 |
| 6 | Eval 与诊断 | ★★★ 强 | S3 | 能定义标准、搭测试、看失败样本、做回归。三层评测（reasoning 15c/49t + E2E 15c + LLM-as-Judge） |
| 7 | 项目推进 | ★★ 中 | S2 | 端到端推 0→1 尚未独立完成过 |
| 8 | 商业理解 | ★ 旁听为主 | S1 | 参与商业模式讨论，未独立做出经过市场验证的商业判断 |

**已超出 S1 的结构性证据**：
1. 参与价值验证，不只做需求实现
2. 比较技术路线并对应业务场景
3. 定义评测标准和回归验证逻辑
4. 主动规划 v3 产品方向并完成完整技术架构对比（非被动执行）
5. 作为产品初始用户，对业务有双重视角（用户 + 产品）

**向 S3 进发的四个差距**：
1. **用户理解**：关键洞察仍高度依赖毛老师框架输入
2. **商业理解**：尚未独立做出经过市场验证的商业判断
3. **迁移性**：最强能力集中在问问大象一条线，未跨场景验证
4. **用户闭环**：用户规模、持续迭代和经营结果尚未出现

**值得单独标记的 meta-skill**

除了八个具体维度，有一个底层模式：**你在一个没有现成路径的领域里，展示了从零建方法论的能力。**

- v2 的评测体系不是"从教程学来然后套用"——这个方向没有成熟教程可抄。你从零定义了三层评测结构（reasoning 层 / E2E 层 / LLM-as-Judge），这是方法论构建能力，不是执行力。
- v3 的架构对比不是"搜索哪个框架好"——你逐行读了两个项目的源码（nanobot v0.1.4.post6 + CowAgent 2.0.4），然后用产品需求做判断标准输出了三套方案和推荐决策。

这个 meta-skill 目前只有一个样本（问问大象），所以还不能说它是稳定的。v3 Nosce 是第一次真正的复现测试——如果在全新的产品形态和技术约束下，你再次独立建出有效方法论，它就从"可能是偶然"变成"可复现的个人优势"。

### 三个诚实问题

1. **哪些是旁观了但没内化的？** 参与了商业模式讨论，但独立从零构建一个定价逻辑，你能做到什么程度？
2. **哪些是平台势能而不是自己的能力？** 毛老师给的框架、方向判断、反馈质量——离开他之后你能靠自己复现多少？
3. **方法论构建能力是稳定的还是偶然的？** v3 Nosce 是验证点——如果过程中高度依赖毛老师的框架输入，那 v2 评测体系可能也有更多他的影子。
